Was ist FitFridge?
FitFridge ist ein Uni-Projekt im Modul „Strukturierte Programmierung" (SoSe26). Es ist eine Web-App, die einen virtuellen Kühlschrank verwaltet. Der Benutzer kann Produkte über einen Barcode oder Produktnamen hinzufügen (die Daten werden automatisch von der OpenFoodFacts-API geholt), die aktuelle Restmenge tracken, und die Nährwerte des Kühlschrank-Inhalts auf einen Blick sehen. Später sollen auch Tagesziele für Kalorien und Makronährwerte eingerichtet werden können, damit der User sein Essverhalten verfolgen kann.
Das Team besteht aus drei Personen: Raam macht das Backend (Service-Logik und Berechnungen), Tung kümmert sich um Datenbank, OpenFoodFacts-API-Anbindung und die Repository-Schicht, und Mika baut das Frontend mit HTML, CSS und Jinja2-Templates. Der Code liegt auf github.com/tungp2302/FitFridge. Die Tech-Stack ist Python 3.13 mit Flask, SQLite als Datenbank, und Pytest für Tests.
Die aktuelle aktive Codebasis liegt im Ordner flaskr_new/. Der alte flaskr/-Ordner enthält Legacy-Code aus einer früheren Architektur und sollte irgendwann gelöscht werden.

flaskr_new/calculations.py (committed 15.05.)
Diese Datei stellt zwei allgemeine Hilfsfunktionen bereit. Die Funktion to_percentage(part, total) berechnet den prozentualen Anteil von part an total und gibt einen Float zurück. Sie fängt den Edge Case ab, dass total gleich Null ist, und gibt in diesem Fall 0.0 zurück, um Division-by-Zero-Fehler zu vermeiden. Die zweite Funktion ist convert_units(amount, from_unit, to_unit), die zwischen Gewichts-Einheiten (mg, g, kg) oder zwischen Volumen-Einheiten (ml, cl, l) konvertieren kann. Wenn versucht wird, zwischen Kategorien zu konvertieren (z.B. Gramm zu Milliliter, was ohne Dichte nicht eindeutig ist), wirft die Funktion einen ValueError mit aussagekräftiger Fehlermeldung.
flaskr_new/nutrition_service.py (committed 19.05.)
Dieses Modul kümmert sich um alle Nährwert-Berechnungen und nutzt intern calculations.py. Es enthält drei Funktionen.
calculate_for_amount(product, amount, unit) rechnet die Pro-100g-Nährwerte eines Produkts auf eine konkrete Menge um. Sie erwartet als ersten Parameter ein Dictionary mit den Schlüsseln kcal_per_100g, protein_per_100g, fat_per_100g und carbs_per_100g – also genau das Format, das aus Tungs product_repo oder fridge_repo kommt. Die Funktion konvertiert die Menge automatisch zu Gramm (kg, mg) und nimmt für Volumen die Vereinfachung an, dass 1ml einem Gramm entspricht (was für Wasser, Milch und ähnliche Flüssigkeiten ausreichend genau ist). Sie hat fünf Edge Cases eingebaut: Wenn die Menge None, kleiner gleich Null, in "stk" gemessen, oder in einer unbekannten Einheit angegeben ist, gibt sie ein Dictionary mit allen Werten auf 0.0 zurück, statt zu crashen. Der Rückgabewert ist ein Dictionary mit den Schlüsseln kcal, protein, fat, carbs, alle auf eine Nachkommastelle gerundet.
calculate_total_for_fridge(items) summiert die Nährwerte über alle Items im Kühlschrank. Sie erwartet eine Liste von Dictionaries im Format, das fridge_repo.list_items() liefert (also mit den Feldern current_amount, unit, und den *_per_100g-Feldern aus dem JOIN mit der Produkt-Tabelle). Intern ruft sie für jedes Item calculate_for_amount auf und summiert die Ergebnisse. Bei einer leeren Liste gibt sie ein Dictionary mit allen Werten auf 0.0 zurück.
calculate_daily_percentage(consumed, daily_target) berechnet, wie viel Prozent eines Tagesziels für jeden Nährwert erreicht wurde. Sie nimmt zwei Dictionaries entgegen, beide mit den Schlüsseln kcal, protein, fat, carbs. Werte über 100 Prozent sind möglich (= Tagesziel überschritten). Wenn das Tagesziel auf 0 gesetzt ist (z.B. weil der User es noch nicht konfiguriert hat), gibt die Funktion 0.0 zurück, anstatt einen Division-Fehler auszulösen. Das funktioniert automatisch, weil sie intern to_percentage aus calculations.py nutzt, die diesen Edge Case bereits behandelt.
Alle Funktionen wurden lokal über eine spielwiese.py-Datei mit insgesamt über 15 Testfällen durchgetestet (Normalfälle und Edge Cases). Die Spielwiese ist nicht im Repository, da sie nur lokal zum Lernen und Testen dient.

Architektur-Hinweis: Doppelte Logik
Es gibt einen kleinen Konflikt zwischen Tungs und meinem Code, der besprochen werden sollte. In fridge_service.py existiert die Funktion calculate_total_nutrition(item), die im Grunde eine eingeschränkte Version meiner calculate_for_amount ist. Tungs Version unterstützt nur die Einheit "g" und gibt bei allem anderen Nullen zurück. Sie hat keine Behandlung für kg, mg, ml, cl, l oder "stk". Die Funktion wird aktuell in routes.py an zwei Stellen verwendet: in der dashboard()-Route und in product_detail().
Mein Vorschlag wäre, dass fridge_service.calculate_total_nutrition entweder zu einem dünnen Wrapper umgebaut wird, der intern nutrition_service.calculate_for_amount aufruft, oder dass routes.py direkt auf nutrition_service umgestellt wird. Letzteres ist sauberer, erfordert aber eine kleine Anpassung in routes.py, weil sich die Schlüsselnamen der Rückgabe-Dicts ändern (von total_kcal zu kcal etc.).

Wie meine Module benutzt werden
Tungs Code kann meine Funktionen sofort nutzen. Die Importe sehen so aus:
pythonfrom flaskr_new.calculations import to_percentage, convert_units
from flaskr_new.nutrition_service import (
    calculate_for_amount,
    calculate_total_for_fridge,
    calculate_daily_percentage,
)
Da alle Funktionen stateless sind, können sie überall aufgerufen werden, ohne dass ein Flask-Context oder eine DB-Verbindung nötig ist. Sie sind reine Berechnungs-Hilfen und werfen keine Exceptions außer convert_units bei inkompatiblen Einheiten (was in der App nicht passieren sollte, weil ich in Service-Aufrufen vorher die Kategorie prüfe).
